Requirements

The requirements we have gathered below have been obtained from: Client interviews, Use cases, UI prototyping

Functional Requirements - MoSCoW [Final]

(square brackets indicate final conditions compare to the initial identified requirements)

Must Have

  • The web app shall display the number of documents ingested. [DONE]
  • The web app shall display the number of risky clusters and documents.
  • The web app shall allow the user to create a new project. [DONE]
  • The web app shall let the user switch between projects on the home tab. [DONE]
  • The web app shall let the user have an overall view of the clusters on the Project Overview tab. [DONE]
  • The web app shall display all the documents in a cluster on the Cluster Overview tab. [DONE]
  • The web app shall allow the user to view the documents. [DONE]
  • The web app shall have simple tabs for user navigation. [DONE]

Should Have

  • The web app shall be able to authenticate users and display projects that are associated to each user correctly. [Canceled because our client mentioned that user authentication should not be a main concern for the app]
  • The web app shall implement a database system, which is used by the second part of the system as well. [Slightly modified as we are now using mock data since the results from second part of the system is not accurate]
  • The web app shall label the risky part of the documents clearly. [DONE]
  • The web app shall allow the user to add in new document to be ingested by the system. [Modified as the documents should be added at the very start of system to go through part I and part II, which are responsible by team 39 and team 40]
  • The web app shall provide two kinds of views in Project Overview, which feature both 3D (Pie Chart) and 2D (Table). [DONE]

Could Have

  • The web app shall allow the user to save a document to read it later. [DONE]
  • The web app shall allow the user to comment on the save-for-later-document for easier reference. [DONE]
  • The web app shall allow the user to give feedback to the system on the risky components. [Canceled by our client because there isn't a perfect way for the system to learn whether the information is correct for all]
  • The web app shall allow the user to open the original document. [DONE]
  • The web app shall allow the user to trace back the clustering of the document. [Modified, done with mock data]
  • The web app shall allow the user to edit the name of the project. [Canceled by our client because he mentioned that the name of the project should not be altered]

Would Have

  • The web app shall implement a more complex admin system, which only certain users can create, edit projects and assign normal users to projects. [Canceled by our client]
  • The web app shall allow document preview on the Saved for Later tab. [DONE]
  • The web app shall allow the user to print the document in both edited and unedited form. [Canceled because it's not useful]
  • The web app shall have pop-up tips of using the web app at the side. [Canceled because our client wants a simple app]

Non-Functional Requirements

  • The web app shall be colour coded and simple to use. [DONE]
  • The web app shall authenticate user in three seconds or less. [Modified, since user authentication is not required]
  • The web app shall offer real time feedback, that is, able to show the changes made by user right after editing. [DONE]
  • The web app shall use font that is elegant and easy to read. [DONE]
  • The web app shall be available 24 hours a day, 7 days a week. [Due to confidentiality, the app is not open to public.]

Use Cases

Use Case NewUser
ID 1
Brief Description A user requests a new account.
Primary Actors User
Secondary Actors None
Preconditions User is not logged in, does not have an account
Main Flow 1. The user clicks on register an account on login page.
2. System generates a detail form to be filled in by the user.
3. User (lawyer) fills in details and submit.
4. System records the data in database.
5. New user registered.
Post conditions 1. System redirects to Login Page.
2. User can now login.
Alternative flows Failed to register new account, error message(s) shown on same page
A) Duplicate email.
B) Field(s) left blank.
Use Case Login
ID 2
Brief Description User attempts to login.
Primary Actors User
Secondary Actors None
Preconditions User not logged in, owns an account
Main Flow 1. User types in Username and Password.
2. User clicks on login button.
3. System checks if the Username and Password are matched or exist.
4. If Username and Password matched, system authenticate user.
Post conditions 1. System redirects user to Home Page.
2. User is logged in.
Alternative flows Invalid Login, error message shown on same page "Invalid Login, please login using registered email address and password, or register"
Use Case CreateNewProject
ID 3
Brief Description User wants to create a new project.
Primary Actors User
Secondary Actors None
Preconditions User is logged in
Main Flow 1. User clicks on 'New Project' button to create new project.
2. System generates a new project form to be filled in by the user.
3. User fills in details and clicks submit.
4. The system records the data in database.
5. New project created.
Post conditions New project is created and is available to view from project tabs.
Alternative flows None
Use Case AddDocuments
ID 4
Brief Description User wants to add new documents into the project.
Primary Actors User
Secondary Actors None
Preconditions 1. User is logged in.
2. A project is created
Main Flow 1. On either Home Page or Project Overview Page, User clicks on 'add document' button to add new documents.
2. A pop-up windows appears, showing system directory for user to select files in certain format.
3. User select files and click done.
4. System ingests the documents and analyse them.
5. The new documents are now grouped according to their clusters.
Post conditions 1. New project now contains more documents.
2. New cluster might be created.
Alternative flows None
Use Case ViewingAnalysisResult
ID 5
Brief Description User wants to have an overall view of the results.
Primary Actors User
Secondary Actors None
Preconditions 1. User is logged in.
2. Documents have been analysed.
Main Flow 1. User clicks on Project Overview Page.
2. System displays Clusters in form of Pie chart to allow users to have an overall idea of the main cluster.
3. User hovers the mouse over one of the clusters.
4. The big cluster then breaks down into smaller clusters, which allow user to click on it to view the documents under that particular cluster.
Post conditions User is able to view the documents with risk analysis.
Alternative flows None
